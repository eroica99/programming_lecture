1. 소프트웨어 생명 주기 모형

소프트웨어 생명 주기란
p.172
소프트웨어 생명 주기 (Software Life Cycle)
소프트웨어를 효과적으로 정의, 개발, 유지·보수하기 위한 단계별 절차이다. 개발 상황에 따라 가장 적합한 모형을 선택하여 적용할 수 있다.
3가지 생명 주기 모형
pp.172–175
① 폭포수 모형 (Waterfall Model)

단계별로 차례대로 진행되는 구조이다. 각 단계는 그 전 단계가 완전히 종료된 후 시작되며, 각 단계에서 구체적인 결과물이 생성된다.

요구 분석→ 설계→ 구현→ 테스트→ 유지·보수
구분내용
장점가장 폭넓게 사용되는 전통적인 모형으로, 단계별 정의와 내용이 분명하고 각 단계의 산출물이 명확하다.
단점개발 중 발생하는 새로운 요구나 조건을 반영하기 어렵다. 초기 단계에서 요구 사항이 명확하게 제시되어야 한다. 실제로 오류 없이 다음 단계로 진행하기 매우 어렵다.
② 프로토타입 모형 (Prototyping Model)

사용자의 요구 사항을 정확히 파악하기 위하여 끊임없이 견본(prototype)을 만들어 수정·보완하면서 최종 소프트웨어를 형성하는 모형이다.

개발 단계에서 사용자의 피드백을 받아 지속적인 유지·보수가 이루어진다.

구분내용
장점요구 사항의 변경이 용이하며, 최종 결과물이 만들어지기 전에 일부 또는 견본을 미리 볼 수 있다.
단점빠른 시간 내에 제작해야 하므로 비효율적인 알고리즘이나 프로그래밍 언어가 사용될 수 있다.
③ 나선형 모형 (Spiral Model)

폭포수 모형과 프로토타입 모형의 장점에 위험 분석 기능을 추가한 모형이다. 나선을 따라 돌듯이 프로토타입을 지속적으로 발전시켜 최종 소프트웨어를 개발하며, 개발 단계에서 발생할 수 있는 위험을 관리하고 최소화하는 것을 목표로 한다.

계획 및 정의→ 위험 분석→ 공학적 개발→ 고객 평가→ 반복
구분내용
장점대규모 프로젝트에 적합하다. 누락되거나 새로운 요구 사항을 반영할 수 있으며, 특별한 유지·보수 과정이 필요 없다. 완성도 높은 소프트웨어를 만들 수 있다.
단점위험을 발견하지 않으면 반드시 문제가 발생한다.
3가지 모형 비교
pp.172–175

핵심 특징단계별 순차 진행견본 반복 제작·수정위험 분석 반복

2. 프로그램 기획
교과서 pp.175–179
요구 사항 분석 과정
pp.175–177

소프트웨어 개발의 첫 단계는 사용자의 요구 분석이다. 이 단계에서 고객이나 사용자의 필요성과 문제점을 파악하여 개발 목적을 설정한다.

요구 추출→ 도메인 분석→ 모델링→ 프로토타이핑과 시험→ 문서화 및 검토
소프트웨어 요구 사항 명세서
pp.176–177
소프트웨어 요구 사항 명세서 (SRS)
소프트웨어 개발의 요구 사항 분석 단계에서 작성되는 최종 산출물이다. 소프트웨어가 수행해야 할 모든 기능과 제약 조건 등이 기술되어 있으며, 이후 진행하는 소프트웨어 개발과 테스트의 완료 조건으로 사용된다.

분석된 요구 사항을 명확하고 완전하게 기록해야 하며, 개발자와 사용자가 합의한 내용을 이해하기 쉽도록 간결하게 작성해야 한다.

프로그램 기획서의 구성 요소
pp.178–179

소프트웨어 요구 사항 분석을 바탕으로 다음 항목을 포함한 프로그램 기획서를 작성한다.

항목내용
프로그램명개발할 프로그램의 이름

대상 사용자프로그램을 사용할 주요 대상
기능 요구 사항입력 기능, 처리(계산) 기능, 결과 표시 기능 등
사용자 인터페이스메인 화면 구성, 입력 필드, 결과 표시 영역 등
3. 프로그램 설계
교과서 pp.180–185
개발 설계서
pp.180–184

소프트웨어 개발을 위한 문제 상황이 제시되면, 요구 분석을 통하여 입력 방법·처리 작업·출력 방법을 명확하게 파악하여 개발 설계서를 작성한다.

설계 항목내용
입력 설계 사용자가 컴퓨터에 어떤 방식으로 데이터를 입력할 것인가를 설계한다.
· 입력 데이터 양이 적을 때: 키보드 등으로 직접 입력
· 입력 데이터 양이 많을 때: 파일 입출력 활용
처리 설계 입력받은 데이터를 어떤 방식으로 처리할지 설계한다. 변수 설계, 계산 공식, 조건 분기 등을 구체화한다.
출력 설계 처리 결과를 어떤 형식으로 출력할지 설계한다. 출력 내용과 형식(화면·파일 등)을 결정한다.
알고리즘 작성
pp.183–185

입출력 데이터와 형식이 결정되면, 수행 작업을 컴퓨터에서 수행할 수 있도록 구체적이고 명확한 알고리즘으로 표현한다. 알고리즘은 순서도 또는 의사 코드로 작성한다.

표현 방법특징
순서도도형과 흐름선으로 알고리즘의 흐름을 시각적으로 표현
의사 코드자연어를 코드처럼 작성하여 알고리즘의 논리 구조를 표현
4. 프로그램 구현과 테스트
교과서 pp.185–188
프로그램 구현
pp.185–186

프로그램의 데이터 입력·처리·출력 과정이 설계되면, 다양한 프로그래밍 언어를 이용해 구현한다. 소프트웨어 개발 프로젝트에 적합한 언어를 선택하고, 해당 언어의 문법과 규칙에 따라 작성한다.

구조적 프로그래밍
순차(sequence), 선택(selection), 반복(iteration)의 기본 제어를 결합하고 중첩해 사용하는 프로그램 구현 방법이다. 프로그램의 실행 과정을 논리·체계적으로 표현할 수 있어 이해하기 쉽고, 작성 과정에서의 논리적 오류도 줄일 수 있다.
프로그램 테스트
pp.187–188 [표Ⅲ-3]

프로그램이 개발되면 요구 사항을 만족하는지 검사(test)를 통하여 품질을 확인해야 한다. 오류가 발견되면 문제점을 찾아 수정·보완한다.

테스트 종류내용
단위 테스트
(unit test)		 단위 모듈에 대한 테스트이다.
· 입출력 테스트, 경계값 테스트, 오류 처리 테스트 등
· 개별 함수나 모듈이 정상적으로 동작하는지 확인
통합 테스트
(integration test)		 단위 모듈을 통합한 후의 테스트이다.
· 여러 모듈이 서로 연결되었을 때 정상적으로 동작하는지 확인
· 모듈 간 인터페이스, 데이터 흐름 등을 검증
시스템 테스트
(system test)		 완성된 전체 시스템에 대한 테스트이다.
· 실제 운영 환경에서 시스템 전체가 요구 사항을 만족하는지 확인
· 성능, 보안, 신뢰성 등 전반적인 품질 검증
